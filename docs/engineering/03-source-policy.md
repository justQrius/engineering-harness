# Source Policy

Use sources in a strict hierarchy so work stays current and grounded.

## Source Hierarchy

1. local docs, code, tests, and decisions
2. official docs, specs, repos, and examples
3. implementation-grounding repos and maintained references
4. broader market or synthesis sources

## Rules

- use at least one internal source for non-trivial work
- use primary external sources when external behavior matters
- refresh fast-moving assumptions
- record important sources in packets or notes
